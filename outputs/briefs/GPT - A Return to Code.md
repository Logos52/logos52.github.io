# GPT - A Return to Code

## Core Thesis

Naval's argument is that AI coding agents make software feel personal again. Coding returns not because everyone wants to become a traditional programmer, but because custom software becomes cheap enough to build for individual needs, small workflows, and private use cases.

The result is a "personal app store": instead of waiting for companies to build general tools, individuals can generate narrow tools for themselves.

## Compressed Takeaways

1. **AI brings people back to code.** Not necessarily to hand-written implementation, but to software creation.
2. **Personal software becomes viable.** Tiny apps for one person or one workflow become worth building.
3. **Vibe coding feels like a game with rewards.** Iteration is fast, feedback is immediate, and the output can be useful.
4. **Pure software may be harder to invest in.** If software gets easier to build, defensibility shifts elsewhere.
5. **Each model has a place.** Different models serve different parts of the workflow.
6. **AI tries to please.** Human judgment is needed because models will satisfy instructions even when the premise is weak.
7. **Math and coding matter because they are verifiable.** AI advances fastest where correctness can be checked.
8. **Interfaces will shift.** Agents may become the front line for many software and service interactions.

## The Personal App Store

The most important idea is that AI makes small, custom applications practical. A workflow that would not justify a startup, SaaS product, or engineering sprint may still justify a personal app.

Examples:

- a private dashboard,
- a one-off file converter,
- a custom note processor,
- a local search tool,
- a personal automation,
- a study tracker,
- a small interface over existing data.

This applies directly to this repo. The knowledge base should not only contain notes. It can accumulate tools, outputs, and workflows that serve the user's actual life.

## Vibe Coding as Play

The source treats vibe coding as motivating because it creates a tight feedback loop:

1. Describe a feature.
2. See something appear.
3. React to it.
4. Ask for changes.
5. Improve it.

This is why it can feel like a video game. The reward loop is visible and useful. But that same speed can also encourage shallow acceptance. Working is not the same as correct, maintainable, or secure.

This links to [[wiki/Concepts/Vibe Coding|Vibe Coding]] and [[wiki/Concepts/Agentic Engineering|Agentic Engineering]]. Vibe coding supplies momentum. Agentic engineering supplies standards.

## Models as Collaborators

Naval's framing implies that different models occupy different roles. Some are better at code. Some are better at writing. Some are better at search, synthesis, or style.

The workflow implication:

- choose the model for the job,
- do not assume one model is globally best,
- use disagreement between models as signal,
- keep the human in charge of final judgment.

This connects to [[wiki/Concepts/LLM Tool Use|LLM Tool Use]] and [[wiki/Red Team/Applied Critical Thinking - Testing Frames|Applied Critical Thinking: Testing Frames]].

## Verifiability and Code

The source emphasizes why coding and math are strong AI domains: outputs can be checked. Code can run. Tests can pass or fail. Errors can be inspected.

This creates a practical standard:

- For code, use tests and builds.
- For writing, use audience, clarity, and argument checks.
- For notes, use coherence, backlink integrity, and source traceability.
- For decisions, use predictions, assumptions, and postmortems.

Where verification is weak, the human needs more skepticism.
